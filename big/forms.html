Interacting with the user: going down the rabbit hole with HTML Forms

HTML forms may seem daunting due to their versatility, their diverse amount of native widgets and all of the associated attributes, not to mention the challenge of form styling with CSS. However, if HTML forms disappeared, there would be no logins, no discussion forum comments, no entering credit card information to complete a purchase, or any interaction between users and website owners. As designers and developers, it's our duty to consider how users may get confused or irritated by our forms and take the necessary steps to ensure smooth completion.

Providing clear instructions that emphasize mandatory fields, labeling all input fields and form controls with the <label> element as well as grouping similar controls together makes a form not only more accessible, but also more intuitive for all users (WCAG WG). The <fieldset> and <legend> elements can bring together radio buttons, checkboxes or any number of topically related widgets. However, it is possible that some screen readers may not read out the <legend> text, thus dropping vital information. Adding the group’s name to the label for the first form control and using CSS to hide that group name visually is a quick fix - see the code snippet below (WCAG WG). Outside of form specific elements, structuring forms with headings (<h1> - <h6>) and sectioning elements (<section>, <div>) is a standard practice that improves usability while also increasing styling opportunities (MDN).

Before spending time dreaming up a unique aesthetic design for a form, it’s worth remembering that users may not be viewing the website on an updated browser and that widget appearance depends on both the browser type and version as well as the operating system (MDN). Beyond the hurdle of legacy browsers, not all form elements can even be styled with CSS. The Mozilla Developer Network provides useful tutorials on how to circumvent these obstacles, but generally the <form>, <fieldset>, <label>, <output> and text field widgets accept CSS styling with few issues across all platforms. Another helpful warning comes from CSS-Tricks author, Chris Coyier: default browser styling applied to widgets can also be broken if a CSS reset is used (Coyier). “Because HTML forms involves complex interaction, there is one rule of thumb: keep it simple, also known as the 'KISS principal'. There are so many cases where we want forms that are 'nicer' or 'with advanced functionality', but building efficient HTML Forms is not a question of design or technology. Rather, it's about simplicity, intuitiveness, and ease of user interaction” (MDN).

The above mentioned points just graze the surface of form authoring - what if your team is short on time? Even Chris Coyier acknowledges that there is a time and a place for “not reinventing the wheel” and recommends Wufoo for custom online form templates and inspiration (Coyier).

Code Snippet to address <legend> accessiblity:

<fieldset>
  <legend>Shipping Address:<legend>
    <div>
        <label for="shipping_name">
        <span class="visuallyhidden">Shipping</span>Name: </label><br>
        <input type="text" name="shipping_name" id="shipping_name">
    </div>
    <div>
        <label for="shipping_street">Street:</label><br>
        <input type="text" name="shipping_street" id="shipping_street">
    </div>
</fieldset>
